bzoj3105 [cqoi2013]新Nim游戏

发表于 2018-05-12 | 分类于 算法竞赛进阶指南 |

中文题面

Description
传统的Nim游戏是这样的：有一些火柴堆，每堆都有若干根火柴（不同堆的火柴数量可以不同）。两个游戏者轮流操作，每次可以选一个火柴堆拿走若干根火柴。可以只拿一根，也可以拿走整堆火柴，但不能同时从超过一堆火柴中拿。拿走最后一根火柴的游戏者胜利。
本题的游戏稍微有些不同：在第一个回合中，第一个游戏者可以直接拿走若干个整堆的火柴。可以一堆都不拿，但不可以全部拿走。第二回合也一样，第二个游戏者也有这样一次机会。从第三个回合（又轮到第一个游戏者）开始，规则和Nim游戏一样。
如果你先拿，怎样才能保证获胜？如果可以获胜的话，还要让第一回合拿的火柴总数尽量小。

Input
第一行为整数k。即火柴堆数。第二行包含k个不超过109的正整数，即各堆的火柴个数。

Output

输出第一回合拿的火柴数目的最小值。如果不能保证取胜，输出-1。
Sample Input
6

5 5 6 6 5 5

Sample Output
21
HINT
k<=100

提示

  在普通的nim游戏中，先手必胜的条件是什么？

代码

分析

  尽量让先手取胜. 若第一轮操作后，存在若干堆石子，使得它们的数量的异或和为0，则先手必输. 因为后手可以取出其它的石子使得异或和变为0，则变成必输局面. 线性基中，无论取多少个数异或起来都不会等于0，故求出石子个数最多的线性基即可.按照石子个数从大到小排序，动态维护线性基即可.

解一个模线性方程组.

代码

分析

  解方程组一般都是用高斯消元解. 在模意义下高斯消元其实和在一般情况下高斯消元是差不多的。区别就是多一个取模运算，要考虑负数的情况. 在消元完后判断解的情况. 最后还原. 还原实际上就是在解一个模线性方程，利用扩展欧几里得算法求一组等于gcd的特解，然后求出最小正整数通解即可.
  这道题使我又掌握了一种新的高斯消元的模板. 以前写过3类模板：
  1.一定有解且仅有一组解的方程组.直接消就好了.
  2.贪心+高斯消元，不改变方程的位置,类似线性基的插入操作.
  3.不一定有且仅有一组解的xor方程组. 每次找到最大的那个数，找到其最高位，将其它的这一位为1的数与之异或即可.
  现在不是xor方程组，而且解的情况可能不唯一，于是换上一种新的模板：记录当前扫到了哪一行，扫到了哪一列. 不管扫到哪一列，一定要将当前行给消掉。为了保证方程组的系数都是整数，乘到最小公倍数来消元.

翻译

求一个矩阵的等比数列和.

分析

  求等比数列可以利用公式，也可以分治. 在本题中采用分治的方式. 一些问题需要注意：
  1.尽量减少不必要的取模.
  2.能不用long long就尽量不用long long.
  3.读入了数据后要取模.
  有了这3个优化，程序就能在规定时间内跑完啦. 如果取模只涉及到加减运算，那么就可以将取模操作变成减法操作. 用long long会非常耗费时间.

In each turn the player can cut either horizontally or vertically, keeping every grids unbroken. After N turns the paper will be broken into N+1 pieces, and in the later turn the players can choose any piece to cut. If one player cuts out a piece of paper with a single grid, he wins the game. If these two people are both quite clear, you should write a problem to tell whether the one who cut first can win or not.

4 2

LOSE

Source

翻译

一张n*m的矩形网格纸，两名玩家轮流行动。在每一次行动中，可以任选一张矩形网格纸，沿着某一行或者某一列的格线，把它剪成两部分。首先剪出1*1的玩家获胜.问先手能否必胜.

代码

分析

  很显然是要用sg来做. 但是这道题和一般的博弈论的题不太一样. 这道题是谁不能动谁就赢了，而利用sg函数求解的博弈论的题要求谁不能动谁就输了. 需要对问题进行转化.
  遇到这种情况，我们先找到必败的边界状态，将其转化为题目的边界状态.比如这道题中，若剪成了2*2,2*3,3*2,那么接下来行动的人必败. 那么这三种情况的sg值为0.
  如何算sg值？ 根据原则多游戏异或，后继求mex,即$sg(n,m) = mex(\{sg(i,m) \^ sg(n-i,m)\} U \{sg(n,i) \^ sg(n,m - i)\})$.
  这么做可能会有问题：在dfs计算sg值的时候可能会走到边界1,1,这样就会得到错误的答案. 为了防止这种情况发生，强制让dfs走到新边界而不是原边界,只需要对决策做一点点改动即可.

bzoj3054 Rainbow的信号

发表于 2018-05-10 | 分类于 算法竞赛进阶指南 |

中文题面

Description
Freda发明了传呼机之后，rainbow进一步改进了传呼机发送信息所使用的信号。由于现在是数字、信息时代，rainbow发明的信号用N个自然数表示。为了避免两个人的对话被大坏蛋VariantF偷听T_T，rainbow把对话分成A、B、C三部分，分别用a、b、c三个密码加密。现在Freda接到了rainbow的信息，她的首要工作就是解密。Freda了解到，这三部分的密码计算方式如下：
在1~N这N个数中，等概率地选取两个数l、r，如果l>r，则交换l、r。把信号中的第l个数到第r个数取出来，构成一个数列P。
A部分对话的密码是数列P的xor和的数学期望值。xor和就是数列P中各个数异或之后得到的数； xor和的期望就是对于所有可能选取的l、r，所得到的数列的xor和的平均数。
B部分对话的密码是数列P的and和的期望，定义类似于xor和。
C部分对话的密码是数列P的or和的期望，定义类似于xor和。

Input
第一行一个正整数N。
第二行N个自然数，表示Freda接到的信号。

Output
一行三个实数，分别表示xor和、and和、or和的期望，四舍五入保留3位小数，相邻两个实数之间用一个空格隔开。

Sample Input
样例输入1
2
4 5
样例输入2
3
1 0 1
Sample Output
样例输出1
2.750 4.250 4.750
样例输出2
0.667 0.222 0.889

提示

  如何求权值和？如何求对应的概率？

代码

分析

  这道题虽然披上了期望的外衣，实质上就是求出所有区间的xor,and,or和.
  涉及二进制运算的问题一般怎么快速求和？ 对每一位分别求和，若第i位有cnt种方案满足其为1，则它对答案的贡献为$2^i * cnt$.将每一位的贡献加起来就是答案了.
  涉及到所有区间的值，通常的方法是固定一个端点，令另一个端点快速移动. and和or运算都具有“单调性”，可以很容易地确定对答案有贡献的区间的个数. 具体方法是：设当前处理的是二进制的第p位，枚举区间的右端点r. 若第r个数的第p位为1，则找到r左边第一个第p位为0的数，记其位置为pos,那么左端点在[pos + 1,r - 1]的区间的and值都对答案有影响. 对于or操作同理.
  比较难处理的是xor操作. 有一个性质：在xor操作中，遇到0异或值不变.遇到1会变成对应的数.那么序列就可以被分成若干段，每一段的端点由1个0和1个1组成. 对这些段进行黑白染色. 若当前的右端点是第r位，那么要么是左端点在黑色段的区间产生贡献，要么是左端点在白色段的区间产生贡献.具体怎么维护见代码.
  上面所提到的区间都没有涉及l = r的情况. 这是因为当l = r时，区间出现的概率是$1 / n^2$,而l != r时，区间出现的概率是$2 / n^2$,故要单独处理l=r的情况.

总结

  xor操作求区间和分成一段一段来维护.

  在DAG上求期望，显然是期望dp. 我一开始想的做法是dfs,到点u后枚举其后继节点v. 若u的出度为k,则$f_v += (f_u + w) / k$. 这种做法有问题，因为dfs是一条道走到黑，可能点v的f值还没有更新完，就用v去更新其后继节点了. 那么改成拓扑排序，结果还是错的. 因为一个有一个原则反向求期望，正向求概率.
  分析样例中的图，2—>3这条边的权值应该乘上1/2（走这条边的概率），而实际计算中是没有乘的. 这是因为在dp的时候没有考虑走到点u的概率是多少，从而转移默认了走u—>v这条边的概率是1.由此可知：如果正向求期望，会错误地计算很多不合法的状态. 一种解决方法是求出到达每个点的概率，那么这个点出边的边权就要乘上这个概率. 还有一种解决方法是构造反向图，倒着转移. 即假设到达v的状态合法，那么从u转移过去的状态一定合法. 由此可以总结出求期望的一般方法：要么是正向求概率乘权值，要么是倒着求期望.倒过来求要注意状态和问题的转化.
